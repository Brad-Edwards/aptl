### Added

**Comprehensive red-side behavioural capture (OBS-003 / ADR-033).**
The Kali container now ships an OBS-003 capture surface — `auditd`
with an APTL ruleset (execve / connect / file ops on /home/kali,
/tmp, /root, /etc), `accton` process accounting, per-session
`script --log-io --return --log-timing` recording (full input+output
PTY with timing, command exit status preserved), and per-session
`tcpdump` with a rolling 1GB cap — wired in through an sshd
`ForceCommand` shell wrapper. Captures land in a docker named volume
`kali_captures:/var/log/aptl/captures` and the MCP-red server
harvests them out via `docker cp` on session close into
`.aptl/runs/<run_id>/kali-side/<session_id>/` on the host with 0600
permissions. The container also gets `AUDIT_CONTROL` / `AUDIT_WRITE`
only at entrypoint time; `CAP_AUDIT_CONTROL` is dropped via `capsh`
before sshd accepts connections so the kali user (passwordless sudo)
cannot disable the audit trail mid-scenario.

On the MCP side, `mcp/aptl-mcp-common/src/ssh.ts` now (a) opens
every SSH session with `SendEnv APTL_SESSION_ID APTL_RUN_ID
APTL_TRACE_ID` for cross-side correlation, and (b) runs a
continuous PTY tee that writes every byte read from the SSH PTY
into `.aptl/runs/<run_id>/mcp-side/sessions/<session_id>.jsonl`,
closing a chunk-loss gap where bytes arriving between caller polls
had no independent record. Tool-call records and OCSF activity
events both land in the same per-run directory.

`src/aptl/core/runstore.py` is extended with session-scoped
directory helpers (`mcp_side_dir`, `kali_side_session_dir`,
`mcp_session_jsonl`) and a trace-context-driven resolver
(`resolve_active_run_dir`); the TypeScript module
`mcp/aptl-mcp-common/src/runs.ts` mirrors the contract on the MCP
side. `trace_id` (already generated by `aptl scenario start`) is
the cross-process correlation key — no new fields added to
`trace-context.json`.
