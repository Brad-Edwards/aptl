**Captured the TechVault kali container ACES inventory.** Added the kali
red-team container steady-state evidence bundle, mapping ledger, and README
under `docs/aces/inventory/kali/`, the TechVault ACES SDL `kali` node (plus the
`redteam-net` network), parity inventory rows, and structural tests for
SCN-010 issue #339. The capture rebuilt the `kali` image from current
`containers/kali/` source and recreated the container before capture. The SDL
encodes the kali node's image identity and build provenance, network
realization, SSH service, runtime mounts, filesystem inventory, process set,
environment, container-wide capability policy, container host/security config
including `extra_hosts`, full dpkg package inventory, Grype scanner findings,
and local identity database. Four steady-state facts that current ACES grammar
cannot express — Docker `init`, `seccomp`/`security_opt`, per-process
capability drop, and SSH `ForceCommand`/`AcceptEnv` — are recorded as
`blocked_by_aces_gap` and tracked upstream as `Brad-Edwards/aces` issues #384,
#385, #386, and #387; issue #339 stays open until those land. Grype is the
primary vulnerability scanner because Trivy has no advisory database for the
`kali-rolling` release.
