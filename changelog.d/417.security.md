### Security

**Stopped mounting the operator lab SSH private key into the vulnerable target containers.**
The `victim`, `workstation`, and `reverse` targets previously bind-mounted the
entire host `./keys` directory, so the operator lab-access private key
(`/keys/aptl_lab_key`) was readable inside the deliberately-vulnerable target
boxes and was captured verbatim into committed ACES inventory evidence. The
root-cause fix splits the keys by purpose instead of redacting the leak: the
operator control-plane key (`aptl_lab_key`) is now a host-only key that is
never mounted into any scenario container, and a dedicated ed25519 pivot key
(`kali_pivot_key`, generated into `config/lab-ssh/` at lab start and
gitignored) carries the kali red box's SSH access into the targets. The
targets now mount only public material — the control-plane public key and the
pivot public key — and their entrypoint authorizes both, so the kali node
pivots with a key that is scenario content rather than the operator's
credential. The three target inventories were re-captured against the rebuilt
images; their `filesystem-sensitive-paths.txt` bundles now contain only public
keys under `/keys`, with the pivot public key inventoried in full as scenario
content. Operators who reused the previously-committed `aptl_lab_key` should
rotate it.
