**`aptl web serve` now ships the GUI and API from a single loopback origin (FastAPI BFF) with a real session credential.**
The web GUI is built as a static SPA and mounted by the FastAPI app, so the
browser, REST/SSE, and the terminal WebSocket all share one origin — no
SvelteKit server hook and no API token in page code. The FastAPI
backend-for-frontend owns the whole control-plane boundary with distinct
concerns: **authentication** is a server-issued two-factor session credential
(Jupyter-style: `aptl web serve` prints a one-time launch URL the operator opens
to bootstrap the session — loopback binding and forgeable `Sec-Fetch-*`/`Origin`
headers are not treated as authentication). The two factors are an
`HttpOnly`/`SameSite=Strict` cookie plus a port-scoped header token the SPA keeps
in `sessionStorage` and echoes as `X-APTL-Session`; both are required for the BFF
to inject the API bearer. This closes the cross-port cookie leak — cookies are
host-scoped, not port-scoped, so a sibling process on another `127.0.0.1:<port>`
could otherwise steal a cookie-only session. **CSRF** defence is the
origin/Fetch-Metadata gate on mutating routes; a `Host` allow-list defends
against DNS rebinding; and a strict CSP (plus `X-Frame-Options`/`nosniff`) hardens
the XSS-readable header-token factor. Terminal WebSocket auth uses short-lived
single-use tickets; the SSE stream uses `fetch` streaming so it can carry the
session header. ADR-039's loopback default and auth-before-route-logic are
preserved. If no GUI build is present, `aptl web serve` fails fast unless
`--api-only` is passed. The split `aptl-web-api` + `aptl-web-ui` Compose profile
remains for dev/preview as a static SPA behind a same-origin reverse proxy, with
the control-plane token held only by the API container; that container declares
the browser-facing origin via `APTL_WEB_PUBLIC_ORIGIN` (`--public-origin`) so the
printed login URL points at the UI origin the SPA runs on, not the API bind
address.
