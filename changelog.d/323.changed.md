**Removed container healthcheck `conditions` from the ACES TechVault scenario.**
The 13 readiness probes (`pg_isready`, `curl`, `dig SOA`, `smbclient`, …) were
authored into the scenario-level `conditions` section, which ACES classifies as
**evaluation** content (the `conditions -> metrics -> evaluations -> TLOs ->
goals` scoring pipeline assessed by an evaluator). Container readiness is a
provisioning concern, not participant-performance scoring, so this made the
scenario spuriously require an evaluator and blocked the `provisioning-only`
ACES backend from realizing it. The healthchecks remain enforced by the
`docker-compose.yml` healthcheck blocks and node-level `runtime.health` status;
only the mismodeled evaluation `conditions` (and their node references) are
removed, so the public ACES start path can boot TechVault under
`provisioning-only`.
